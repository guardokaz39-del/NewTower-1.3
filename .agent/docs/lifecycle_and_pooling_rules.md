# ♻️ Lifecycle and Pooling Rules

В NewTower (из-за ограничений Garbage Collector'а в браузерах) мы активно применяем паттерн **Object Pooling**. Это означает объекты, которые часто создаются и уничтожаются (пули, враги, тексты урона, частицы), **не удаляются через `new` и сборку мусора**. Они переиспользуются.

## Главный Принцип: Zero-Allocation в Hot-Paths

В функциях `update()`, `draw()`, `takeDamage()` строго запрещено использовать ключевое слово `new` для создания физических объектов, массивов или объектов `{ x, y }`.
Любое выделение памяти (allocation) на каждый кадр заставляет Garbage Collector "икать", создавая лаг-спайки (GC Spikes).

---

## 🏗️ Жизненный цикл пулируемого объекта (Enemy / Projectile / Particle)

### 1. Конструктор (`constructor`)

Вызывается **один раз** при первом создании пула. Он только выделяет "жесткую" память.
*Здесь создаются фиксированные массивы и нулевые объекты `{ x: 0, y: 0 }`.*

### 2. Инициализация (`init`)

Вызывается, когда объект достают из пула для работы.
Здесь перезаписываются **только примитивы** (числа, строки, булевые значения).

### 3. Основной Цикл (`update`/`draw`)

Горячий путь (hot path). **Без `new`!** Любая геометрия или вектора считаются через предварительно созданные zero-allocation объекты (напр. `private _moveVector = { x: 0, y: 0 };`).

### 4. Смерть (`die`)

Запуск спецэффектов, выдача наград, вызов `EVENT_DIED`. Установка флага "Готов к возврату в пул" (`hp = 0` или `alive = false`).

### 5. Сброс (`reset`) - 💀 КРИТИЧЕСКИЙ ЭТАП

Вызывается менеджером пула *сразу после возврата объекта в резерв*.
Здесь должна происходить **параноидальная подчистка**. Очищайте абсолютно все поля, таймеры, счетчики стаков, и ID'шники, чтобы следующий объект, который получит этот кусок памяти, не родился с уже висящим на нём Ядом или Щитом.

---

## 🚨 Опасность Неполного `reset()`

Самая частая ошибка - забыть обнулить массивы бафов или таргет-слоты. В итоге "новый" враг рождается и тут же умирает от горения, повешенного на "старого" врага 2 минуты назад.

**Как очищать массивы правильно:**

```typescript
// ❌ Плохо (Allocation - создаст мусор)
this.myBuffs = []; 

// ✅ Хорошо (Перезапись длины массива, O(1) и без аллокаций)
this.myBuffs.length = 0;
```

**Как очищать ссылки:**
Если ваш объект держал ссылку на другой DOM элемент, класс или функцию — обнулите её (`null` или `undefined`), иначе это спровоцирует классическую утечку памяти (Memory Leak), так как пул будет вечно удерживать ссылку, не давая GC ее удалить.

Пример для компонентов UI:

```typescript
public destroy() {
    this.elOverlay.remove();
    // 💀 КРИТИЧЕСКИ ВАЖНО: Разрыв ссылок на DOM
    this.elOverlay = null as any; 
    this.elContent = null as any;
    this.scene = null as any; 
}
```
