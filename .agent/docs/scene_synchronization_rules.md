# Разделение логики: GameScene vs EditorScene

В архитектуре нашего проекта существуют две параллельные, независимые сцены: `GameScene.ts` (Геймплей) и `EditorScene.ts` (Редактор карт).
Это решение изолирует логику игрового цикла (волны, враги, стрельба) от логики маппинга (кисти, undo/redo, сохранение).

Однако, это налагает жесткие технические ограничения на модификацию базовых компонентов (`Map.ts`, `UIManager`), так как они используются (Share) обеими сценами:

## 1. Рендеринг Карты (The Render Pipeline)

Вызов `map.draw(ctx)` не является исчерпывающим завершением рендера карты.

`Map.ts` хранит сетку тайлов, объекты и факелы, но не может самостоятельно применять `Z-Sorting` (сортировку глубины) относительно врагов, так как не знает о них.
Поэтому `Map.ts` отдает инструменты для отрисовки, а **Сцена (Scene)** решает, как их собрать:

**GameScene (Геймплей):**

- Собирает `map.getDynamicRenderables()` (например, анимированные кристаллы).
- Собирает массив врагов.
- Сортирует их по `Y`-координате (Depth Sorting).
- Рисует с учетом Z-индекса.

**EditorScene (Редактор):**

- Не имеет врагов.
- Обязана вручную вызвать `map.getDynamicRenderables()` и отрисовать их поверх статической карты, иначе динамические объекты просто не отобразятся.

> **⚠️ ПРАВИЛО:** При изменении слоев графики (отделение теней, вынос объектов в динамику, добавление декалей), всегда проверяйте, что **и GameScene, и EditorScene** обновлены и вызывают новые методы `Map`, иначе в одной из сцен графика "пропадет".

> **📝 Map State Injection (паттерн из Portal/Base):** Портал и База рендерятся через `getDynamicRenderables()`, поэтому обе сцены получают их автоматически. Реактивное состояние (волны, HP) передаётся через сеттеры Map (`setWaveActive()`, `setLivesRatio()`), подписки на EventBus живут в GameScene.unsubs — EditorScene не затрагивается. При добавлении нового реактивного визуала в Map, **всегда** использовать этот паттерн: примитивное поле + сеттер + подписка через unsubs.

## 2. Обработка Инпута и Состояния (UI Route)

`EditorState.ts` объявляет, в каком состоянии сейчас редактор (`mode: 'paint_grass' | 'paint_fog'`).
В редакторе нет паттерна "State Machine" с полиморфизмом для каждого инструмента. Инструменты обрабатываются процедурно через гигантский роутер `EditorScene.applyToolAt()`.

> **⚠️ ПРАВИЛО:** Добавление переменной в стейт или кнопки в `EditorSidebar` не автоматизирует работу функционала. Необходимо физически прописать ветку поведения `else if (this.state.mode === '...')` внутри `EditorScene.ts`, иначе инструмент станет "кнопкой без действия".

## 3. Waypoint Manager vs AI Navigation

Редактор сохраняет пути используя разреженные, абстрактные маркеры (Waypoints).
Например: Старт (x: 0, y: 0) -> Конец (x: 10, y: 0).

Игровой движок (GameScene) использует непрерывные пути (Fullpath) массив координат тайлов, которые AI применяет для FlowField и поиска кратчайшего маршрута.
`MapData.ts` хранит свойство `waypointsMode: 'FULLPATH' | 'ENDPOINTS'`.

> **⚠️ ПРАВИЛО:** `EditorScene` при экспорте карты обязана резолвить (разворачивать) разреженные точки в непрерывный путь (`resolveFullPath()`) через BFS и сохранять массив каждого тайла, меняя флаг на `FULLPATH`. Если этого не сделать, враги в игре будут телепортироваться между двумя маркерами.
