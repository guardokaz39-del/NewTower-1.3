# ⚠️ Распространённые Ошибки и Ловушки (Common Pitfalls)

Этот документ собирает реальные баги, которые мы уже ловили в проекте. Если вы делаете что-то похожее — проверьте этот список.

---

## 1. 🦴 Скелет без черепа (Rendering)

**Что случилось:** `SkeletonMinerRenderer` рисовал каску, но под ней не было черепа — каска висела на шее.

**Причина:** `BaseSkeletonRenderer.drawTorso()` **НЕ рисует череп** (делегирует подклассам). `drawHeadDecoration()` шахтёра сразу рисовал каску, пропустив `this.drawSkull()`.

**Правило:**

```typescript
// В drawHeadDecoration() наследника BaseSkeletonRenderer:
this.drawSkull(ctx, s, pose.facing);  // ← ВСЕГДА первым!
// ... потом экипировка
```

**Файлы:** `BaseSkeletonRenderer.ts`, `SkeletonMinerRenderer.ts`, `SkeletonUnitRenderer.ts`

---

## 2. 📏 Разреженные waypoints ≠ FULLPATH (Editor)

**Что случилось:** При сохранении карты `validatePath()` выдавал "disconnected" и "loop".

**Причина:** `WaypointManager.getFullPath()` возвращает **разреженные** точки (Start, WP1, WP2, End), не тайл-за-тайлом путь. Если записать их в `map.waypoints` с режимом `'FULLPATH'`, валидация падает — она проверяет Manhattan distance ≤ 1 между соседними точками.

**Правило:** Всегда пропускать через `resolveFullPath()`, который решает каждый сегмент через BFS `Pathfinder.findPath()`.

**Файлы:** `EditorScene.ts` (метод `resolveFullPath()`)

---

## 3. 🏷️ `static readonly` vs Instance Properties (ООП)

**Что случилось:** Дочерние классы не могли менять цвета костей, объявленные как `protected static readonly`.

**Причина:** Статика привязана к классу, а не к экземпляру. `SkeletonMiner` не мог переопределить `BONE_MAIN` из `BaseSkeletonRenderer`.

**Правило:** Для переопределяемых параметров используйте свойства экземпляра:

```typescript
// ❌ Плохо
protected static readonly BONE_MAIN = '#e0d0b0';

// ✅ Хорошо
protected boneMain = '#e0d0b0'; // override в дочернем классе
```

## 4. 🎨 Забытая регистрация в SpriteBaker (Rendering)

**Что случилось:** Новый враг рисовался процедурно каждый кадр → просадка FPS.

**Причина:** Пропущен шаг 5 из [enemy_creation_pipeline.md](enemy_creation_pipeline.md) — вызов `SpriteBaker.bakeWalkCycle()` в `Assets.ts`.

**Правило:** После создания рендерера **ОБЯЗАТЕЛЬНО** добавьте запекание в `generateBakedSprites()`.

## 5. 🦀 "Крабья походка" (IK Animation)

**Что случилось:** Ноги врага расходились при виде спереди/сзади.

**Причина:** Логика `SIDE` IK (смещение ног по X) применялась ко всем `facing` без разделения.

**Правило:** Жёстко разделять IK для `SIDE` (2D IK по X) и `UP/DOWN` (проекционный по Y). См. [procedural_animation_rules.md](procedural_animation_rules.md), раздел 1.

## 6. 📦 `ctx.filter` в горячих путях (Performance)

**Что случилось:** Зависание при большом количестве врагов.

**Причина:** `ctx.filter = 'brightness(2.0)'` для hit flash крайне медленный.

**Правило:** Использовать `ctx.globalAlpha` или pre-baked спрайты вместо `ctx.filter`. См. [enemy_visual_design.md](enemy_visual_design.md), раздел 6.

## 7. 🎆 Динамические градиенты в горячих путях (Performance)

**Что случилось:** При спавне толпы берсерков FPS падал из-за механики Ярости.

**Причина:** В состоянии Ярости (`hp < 50%`) каждый кадр вызывался `ctx.createRadialGradient` внутри `drawEmissive`, создавая огромную нагрузку на GC и растеризатор для каждого из 50 врагов.

**Правило:** Запрещено создавать градиенты на каждом кадре. Используйте заранее запеченные спрайты свечения из `Assets.ts` (например, `Assets.get('fx_glow_red')`) и рисуйте их через `ctx.drawImage`.

## 8. 🏗️ ES2022 Class Field Initialization Order (TypeScript)

**Что случилось:** `WaveTimeline` крашился с `Cannot read properties of undefined (reading 'getBoundingClientRect')`. Поле `private canvas!: HTMLCanvasElement` было `undefined` несмотря на присвоение в `createRootElement()`.

**Причина:** ES2022 `useDefineForClassFields: true` (по умолчанию в современном TS). Порядок выполнения:

1. `super()` (BaseComponent) вызывает `createRootElement()` → `this.canvas = createElement('canvas')` ✅
2. super() завершается
3. **Инициализаторы полей подкласса** запускаются → `canvas!` **перезаписывает** значение на `undefined` ❌

**Правило:** НИКОГДА не объявляйте поле класса, если оно инициализируется в методе, вызванном из `super()`:

```typescript
// ❌ КРАШ: поле перезапишется после super()
class MyComponent extends BaseComponent {
    private canvas!: HTMLCanvasElement;
    protected createRootElement() {
        this.canvas = document.createElement('canvas');
        return this.canvas;
    }
}

// ✅ Используйте this.element (уже установлен BaseComponent)
class MyComponent extends BaseComponent {
    protected createRootElement() {
        const canvas = document.createElement('canvas');
        return canvas;
    }
    render() {
        const canvas = this.element as HTMLCanvasElement;
    }
}
```

**Файлы:** `WaveTimeline.ts`, `BaseComponent.ts`

## 9. 💾 `normalizeWaveConfig()` — Уничтожение данных (Data Layer)

**Что случилось:** Новые поля волн (`name`, `startDelay`, `waitForClear`, `bonusReward`, `shuffleMode`) терялись после каждого save/load.

**Причина:** `normalizeWaveConfig()` в `Utils.ts` строил результат через жёсткий объект `{ enemies: [...] }`, отбрасывая ВСЕ поля, не перечисленные явно. Новые поля из `IWaveConfig` просто не копировались.

**Правило:** При добавлении новых полей в `IWaveConfig` или `IWaveGroupRaw`:

1. Обновите `normalizeWaveConfig()` в `Utils.ts` — должен сохранять и клампить новые поля
2. Обновите `migrateMapData()` в `MapData.ts` — добавьте санитизацию
3. Проверьте round-trip тестом: `normalize(input) → JSON → normalize → assertEqual`

**Файлы:** `Utils.ts`, `MapData.ts`

## 10. 🔄 `oninput` на Range-слайдерах → Флуд undo-стека (UI)

**Что случилось:** Один сдвиг слайдера `baseInterval` генерировал 30–50 записей в undo-стеке, делая Ctrl+Z бесполезным.

**Причина:** `range.oninput` срабатывает на каждый пиксель при перетаскивании, каждый вызов запускал `model.updateWaveSettings()` → `history.push()`.

**Правило:** Для range-слайдеров, подключённых к undo-системе:

- `onchange` — запись в историю (срабатывает при отпускании)
- `oninput` — только визуальное обновление (синхронизация числового поля)

```typescript
// ❌ Плохо: 50 записей за один drag
range.oninput = () => { model.update(range.value); };

// ✅ Хорошо: 1 запись при отпускании
range.onchange = () => { model.update(range.value); };
range.oninput = () => { numInput.value = range.value; }; // Только UI
```

**Файлы:** `SpawnTimingControl.ts`

## 11. 👻 "Невидимые" компоненты (DOM)

**Что случилось:** Добавили `ValidationPanel` в интерфейс редактора волн, и весь список волн (`WaveList`) внезапно исчез, хотя в памяти он создавался исправно. Пользователь видел только кнопку сохранения и пресеты.

**Причина:** При рефакторинге и добавлении нового компонента была случайно удалена строка `this.container.appendChild(content);`, которая прикрепляла контейнер со списком волн к DOM дереву.

**Правило:** В ванильном DOM `mount()` или вставка элемента может не привести к его отображению, если родительский элемент сам не был прикреплен (appended) к документу или главному контейнеру. Всегда проверяйте DOM иерархию после рефакторинга UI.

## 12. 💧 Утечки памяти в UI (Memory Leaks)

**Что случилось:** При закрытии и повторном открытии `WaveEditor` накапливались слушатели событий и не откреплялись старые панели, что приводило к лагам.

**Причина:** Новые панели (`PresetPanel`, `ValidationPanel`) создавались в `WaveEditor`, но не перечислялись в функции `WaveEditor.destroy()`. Метод `destroy()` старых экземпляров не вызывался.

**Правило:** Каждое поле класса в UI-компоненте, которое реализует `destroy()` (или наследует от абстракций с отписками), должно быть явно вызвано в очистке родительского класса:

```typescript
public destroy() {
    if (this.childPanel) this.childPanel.destroy(); // ← НЕ ЗАБУДЬТЕ!
}
```

## 13. 📦 Необработанные ошибки `localStorage` (Storage)

**Что случилось:** Попытка сохранить слишком большой пресет ломала редактор. При плохих данных в `localStorage` (кто-то вручную вписал туда строку вместо JSON массива) ничего не загружалось, и приложение падало при рендере.

**Причина:** Отсутствие защиты от `QuotaExceededError` при `setItem`, отсутствие валидации типа (например `Array.isArray()`) при десериализации.

**Правило:** Работа с `localStorage` **ОБЯЗАНА** иметь блок `try/catch` для чтения и записи. Данные из локального хранилища считаются "не доверенными" (untrusted) — проверяйте их формат перед использованием.

## 14. 🌟 CSS анимации при частых перерисовках (UX)

**Что случилось:** Раздражающее мерцание (flickering) каждого элемента волны при вводе символа в текстовое поле названия волны.

**Причина:** Анимация `animation: we-fade-in 0.2s` применялась к элементу внутри метода `render()`. Метод `render()` вызывался на каждое изменение данных (ввод символа). Соответственно, при каждом `render` элемент создавался заново и CSS анимация проигрывалась снова и снова.

**Правило:** Анимации появления должны применяться к элементу один раз (при монтировании). Если ваш компонент (как `EnemyGroupRow`) часто полностью пересоздаётся через `.innerHTML = ''` при изменении данных, не вешайте на него долгие CSS анимации монтажа.

## 15. 💥 Out-of-bounds индексация (UI/State)

**Что случилось:** Аккордеон списка волн крашился или отображал пустоту при загрузке встроенного пресета.

**Причина:** У нас была открыта волна номер `15` (`expandedWaveIndex = 14`). Пользователь загрузил пресет всего из `3` волн. Локальный стейтUI остался `expandedWaveIndex = 14`, в то время как список `waves` сократился до `3`, и рендер сломался.

**Правило:** При рендере индекс-ориентированных UI-компонентов всегда проверьте целостность индекса относительно текущей длины массива данных.

```typescript
// ✅ Защитный clamp-фактор
if (this.expandedWaveIndex >= waveCount) {
    this.expandedWaveIndex = waveCount > 0 ? waveCount - 1 : -1;
}
```

## 16. 🔤 `sanitizeFileName()` и не-Latin символы (Data)

**Что случилось:** `sanitizeFileName('  Тест 123  ')` возвращал `__123_` вместо `123`.

**Причина:** Цепочка `replace(/\s+/g, '_')` → `replace(/[^a-z0-9_\-]/g, '')` оставляла двойные `_` после удаления кириллицы. Пробелы → `_`, затем кириллица удалена, но `_` от пробелов и `_` от между-символьных позиций сливались.

**Правило:** При линейной цепочке regex-замен, если средние шаги удаляют символы, ВСЕГДА добавляйте финальный коллапс:

```typescript
// ✅ Безопасная цепочка
name.toLowerCase()
    .replace(/\s+/g, '_')
    .replace(/[^a-z0-9_\-]/g, '')
    .replace(/_+/g, '_')       // ← коллапс дублей
    .replace(/^_|_$/g, '')     // ← trim крайних
```

**Файлы:** `MapStorage.ts` → `sanitizeFileName()`

## 17. ⚡ Race Condition при async append в UI (Async)

**Что случилось:** При быстром двойном нажатии на панель карт (toggle → toggle) bundled карты вставлялись дважды.

**Причина:** `refreshList()` вызывала `getBundledMaps().then(append)`. Если вызвать дважды быстро — два промиса запускались параллельно, и оба вставляли карты в DOM.

**Правило:** Для любого UI-метода, который делает async append:

```typescript
private _refreshGeneration = 0;

refreshList() {
    const gen = ++this._refreshGeneration;
    container.innerHTML = '';
    // ... sync render ...
    asyncFetch().then(data => {
        if (gen !== this._refreshGeneration) return; // ← discard stale
        // ... append ...
    });
}
```

**Файлы:** `EditorScene.ts`, `MenuScene.ts`

## 18. 📦 Bundled карты без миграции = тихая поломка (Data)

**Что случилось:** Потенциальный баг — карты в `public/maps/` лежат на старой версии схемы (например, без `fogData`). При загрузке без `migrateMapData()` они вызывали бы крэш в рендере.

**Причина:** Bundled JSON-файлы не проходили через `migrateMapData()`, в отличие от localStorage-данных.

**Правило:** Любые данные из внешнего источника (fetch, File import, localStorage) ОБЯЗАНЫ проходить через `migrateMapData()` + `validateMap()`. Это единственный способ гарантировать совместимость с текущей `IMapData`.

```typescript
// ✅ Правильно: bundled тоже мигрируют
const raw = await fetch(url).then(r => r.json());
const migrated = migrateMapData(raw);
if (!validateMap(migrated)) { skip(); }
```

**Файлы:** `MapStorage.ts` → `_fetchBundledMaps()`, `importFromFile()`

## 19. ⌨️ "Залипание" модификаторов (Input)

**Что случилось:** Инструмент "Пипетка" (Eyedropper) активировался по удержанию клавиши `Alt`. Если пользователь зажимал `Alt` и переключался на другое окно (Alt+Tab), `keyup` событие терялось, и редактор навсегда "застревал" в режиме пипетки.

**Причина:** Браузер не рассылает события клавиатуры, когда окно теряет фокус. Клавиша остается логически "нажатой" в нашем стейте.

**Правило:** Если вы завязываете логику на удержание (Hold) клавиши (`isAltDown = true`), **ВСЕГДА** добавляйте сброс состояния на событие потери фокуса окном:

```typescript
// ✅ Обязательный сброс при blur
window.addEventListener('blur', () => {
    this.isAltDown = false;
    this.isShiftDown = false;
});
```

**Файлы:** `EditorScene.ts`

## 20. 🐢 O(N) поиск объектов в Render Loop (Performance)

**Что случилось:** В ночном режиме `GameScene` искала все факелы на карте, перебирая массив `map.objects` (в котором могут быть сотни деревьев, камней и т.д.) на **каждом кадре** отрисовки.

**Причина:** Логика извлечения координат факелов для системы освещения (`LightingSystem.addLight()`) была скопирована из одноразового `EditorScene.draw` напрямую в Game Loop `GameScene.draw`.

**Правило:** Никаких `array.find` или `array.filter` по большим массивам внутри функции `render()` или `draw()`. Статические (неподвижные) данные, такие как освещение от объектов карты, должны кэшироваться на этапе инициализации (например, в `loadMap` или выделенном `cacheStaticLights()`).

```typescript
// ❌ Плохо: Итерация по всем объектам на каждый кадр (60 FPS)
for (const obj of this.map.objects) {
    if (obj.type === 'torch') this.lighting.addLight(obj.x, obj.y);
}

// ✅ Хорошо: Кэширование статических огней при загрузке
// В цикле draw() мы просто рисуем закэшированный массив this.mapStaticLights
```

**Файлы:** `GameScene.ts`

## 21. 🖌️ Мультипликация сайд-эффектов при Brush Size > 1 (Logic)

**Что случилось:** Рисование "Тумана" кистью размером 3x3 создавало хаотичный визуальный шум: центральный тайл менял плотность один раз, соседний — два раза, крайний — три раза и т.д., пока мышь двигалась.

**Причина:** В режиме `paint_fog` клик по тайлу циклично переключал плотность тумана (0→1→2). Когда кисть `3x3` применяла команду к 9 тайлам, каждый сдвиг мыши заставлял перекрывающиеся тайлы переключаться заново много раз.

**Правило:** Когда инструмент воздействует на *много* тайлов за раз (Brush Tool или Flood Fill), состояние изменения должно вычисляться **один раз** по центральному тайлу (pivot) перед циклом, и применяться как **абсолютное значение** ко всей площади, а не как дельта-вызов к каждому индивидуальному элементу.

```typescript
// ❌ Плохо: Каждый тайл сам решает свою судьбу в кисти
applyToolAt(col, row) {
    this.fog.cycleDensity(col, row); // Вызовется 9 раз по-разному
}

// ✅ Хорошо: Централизованное решение
const targetFog = (this.fog.get(centerCol, centerRow) + 1) % 6;
for(let d = 0; d < 9; d++) {
    applyToolAt(dx, dy, targetFog); // Применяет абсолютное значение
}
```

**Файлы:** `EditorScene.ts`

## 22. 🧟‍♂️ Zombie Event Triggering (Logic/Events)

**Что случилось:** Враг умирал от яда/горения, `ENEMY_DIED` триггерился. На следующем кадре (если враг еще не был удален сборщиком из массива или находился в пуле в процессе очистки), тик DoT-урона применялся снова, здоровье уходило в минус, и событие смерти триггерилось во второй раз.
Это ломало тесты, приводило к двойному начислению золота и дублированию визуальных эффектов смерти.

**Причина:** Отсутствие защиты от применения урона и эффектов к уже мертвым сущностям (hp <= 0). Конвейер урона (`takeDamage`) продолжал выполняться до конца.

**Правило:** Все публичные методы, изменяющие стейт или здоровье сущности (`takeDamage`, `applySlow`, `applyBurn`), должны начинаться со строгой защиты проверки жизнеспособности:

```typescript
// ✅ Обязательный guard в начале метода-мутатора
public takeDamage(amount: number, sourceId: number): void {
```

## 23. 💥 Infinite Recursion & CallStack Overflow (Architecture)

**Что случилось:** Враг с включенной механикой "Взрыв при смерти" (`onDeathExplode`) умирал. При смерти он мгновенно наносил AoE урон соседним врагам через `CollisionSystem`. Если урон убивал соседа (у которого тоже был взрыв при смерти), сосед тут же взрывался изнутри вызова `takeDamage`, убивая третьего. Движок падал с ошибкой `RangeError: Maximum call stack size exceeded`.

**Причина:** Цепные реакции (смерть -> урон -> смерть -> урон) внутри одной синхронной итерации (вложенные вызовы).

**Правило:** Никогда не разрешайте рекурсивные цепочки эффектов (AoE, splash, chain lightning) выполняться синхронно внутри обработчика смерти или урона.
Используйте паттерн **Deferred Event Queue (Очередь отложенных событий)** и ограничение на количество прорубок очереди (Processing Limit):

```typescript
// ❌ Плохо: Синхронный взрыв внутри .die()
die() {
    collisionSystem.applyExplosion(this.x, this.y, 50); // Вызовет takeDamage у соседей
}

// ✅ Хорошо: Отложенная очередь
die() {
    CollisionSystem.queueExplosion(this.x, this.y, 50); // Только сохранение параметров
}

// В основном update() системы
let processLoops = 0;
while (CollisionSystem.explosionQueue.length > 0 && processLoops < 10) {
    this.processExplosions(); // Разбираем статическую очередь
    processLoops++;
}
```

**Файлы:** `CollisionSystem.ts`

## 24. 🧟‍♂️ Охота на Зомби (Object Pooling & Targeting)

**Что случилось:** Башни стреляли сквозь всю карту в сторону точки возрождения врагов, игнорируя свой радиус атаки. Снаряды летели бесконечно долго.

**Причина:** Башня запоминала цель (ссылку на объект `Enemy`). Когда врага убивали, он возвращался в Object Pool (сбрасывая `hp`). Через пару секунд этот же объект доставали для спавна нового врага на краю карты. Башня, которая по таймеру еще не почистила кэш таргета, видела, что `this.target.isAlive() == true` (враг-то воскрес!) и стреляла по нему.

**Правило:** При использовании Object Pooling и кэширования ссылок на активные сущности, **ВСЕГДА** добавляйте физическую валидацию состояния (Range Guard) в каждый кадр, где используется эта ссылка, а не только по таймеру поиска.

```typescript
// ✅ Обязательная проверка физики объекта в пуле каждый кадр
if (this.target && this.target.isAlive()) {
    const distSq = getDistSq(this.target, this);
    if (distSq > this.rangeSquared) {
        this.target = null; // Враг воскрес далеко / вышел из зоны
    }
}
```

**Файлы:** `Tower.ts`, `TargetingSystem.ts`

## 25. 🕳️ Пропуск элементов при удалении (Forward Loop Array Splice)

**Что случилось:** Некоторые снаряды пролетали мимо врагов, некоторые частицы навсегда зависали на экране. Баг проявлялся только при большом скоплении (стресс-тест).

**Причина:** При итерации по массиву с конца в начало программист использовал цикл вперед `for (let i = 0; i < arr.length; i++)` и вырезал элементы через `arr.splice(i, 1)`. При удалении все элементы сдвигались влево. Следующая итерация `i++` **пропускала** элемент, который встал на место удаленного. Попытки чинить через `i--` приводили к трудноуловимым гонкам.

**Правило:** Если в цикле возможно удаление элементов из самого итерируемого массива — **СТРОГО** используйте обратную итерацию (Reverse Loop) или паттерн Swap & Pop.

```typescript
// ❌ Плохо: Пропуск элементов или медленный splice
for (let i = 0; i < effects.length; i++) {
    if (effects[i].isDead) {
        effects.splice(i, 1);
        i--; // Хрупкий хак
    }
}

// ✅ Хорошо: Reverse Loop + Swap Pop (0 сдвигов памяти)
for (let i = effects.length - 1; i >= 0; i--) {
    if (effects[i].isDead) {
        effects[i] = effects[effects.length - 1];
        effects.pop();
    }
}
```

**Файлы:** `ProjectileSystem.ts`, `EffectSystem.ts`

## 26. 🥶 `Object.freeze` и Spread в Hot Paths (Performance/GC)

**Что случилось:** В попытках защитить закэшированный стейт башни от внешних мутаций разработчик добавил `return Object.freeze({ ...this._frameStats })` прямо в метод `getStats()`, который вызывается каждый кадр для каждой башни.
В результате игра начала фризить из-за агрессивной работы Garbage Collector (выделение новых объектов `10 башен * 60 FPS = 600 аллокаций в секунду`).

**Причина:** Оператор спреда `{ ...obj }` создает **новый объект** в памяти каждый раз. Вызов `Object.freeze` поверх него гарантирует, что движок не сможет оптимизировать этот объект (V8 хреново переваривает массовые freeze).

**Правило:** В горячих циклах (Hot Paths) и методах, которые вызываются каждый кадр:

1. НИКОГДА не используйте Spread operator (`...`) или `Object.assign({}, ...)`.
2. НИКОГДА не используйте `Object.freeze` динамически на каждой итерации.
3. Если нужен неизменяемый контракт, используйте TypeScript Utility Type `Readonly<T>` на уровне возвращаемого типа — это защита этапа компиляции, которая не стоит **ничего** в рантайме (Zero-Alloc).

**Файлы:** `Tower.ts` (TIER 2 Caching)

## 29. 💥 Перезапись существующего стейта (Falsy OR Operator)

**Что случилось:** Башня поддерживает ручной выбор `targetingMode` пользователем. При добавлении новой карты характеристики пересчитываются (`getStats()`) и режим цели сбрасывается на дефолтный `'first'`, игнорируя пользовательский ввод.

**Причина:** Использование оператора `||` для слияния конфигураций без проверки того, нужно ли вообще изменять свойство.
`this.targetingMode = mergedMods.targetingMode || 'first';`

**Правило:** При пересчете и применении динамических характеристик из модификаторов/карт к стейту башни, **МЕНЯЙТЕ** стейт только если модификатор явно этого требует.

```typescript
// ❌ Плохо: Всегда перезаписывает текущий стейт (ломает выбор юзера)
this.targetingMode = mergedMods.targetingMode || 'first';

// ✅ Хорошо: Безопасное применение (оставляет текущий если нет оверрайда)
if (mergedMods.targetingMode) {
    this.targetingMode = mergedMods.targetingMode;
}
```

**Файлы:** `Tower.ts` (и любые другие геттеры/сеттеры статов)

## 30. 🧮 Проценты вместо Абсолютных значений в Геймплейной Логике

**Что случилось:** Целеуказатель "Healthiest" (самая здоровая цель) наводил Снайпера на мелких гоблинов (100 HP / 100 HP = 100%), полностью игнорируя раненого Босса (1500 HP / 2000 HP = 75%). Снайпер должен убивать самых "жирных", а не просто здоровых.

**Причина:** Разработчик трактовал `healthiest` как "наивысший процент здоровья: `current / max`".

**Правило:** В игровых механиках выбора целей (Targeting) или сортировки приоритетов (Priority Queues) всегда опирайтесь на **абсолютные значения** (например, `score = e.currentHealth`), если геймдизайн явно не диктует использование процентов (например, для эффектов типа "Добивание: наносит x2 урон врагам с <20% HP"). Абсолютные значения лучше отражают геймплейную ценность/угрозу юнита.

**Файлы:** `TargetingSystem.ts`
