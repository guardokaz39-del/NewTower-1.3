# Руководство по тестированию (Vitest)

В проекте настроена инфраструктура автоматизированного тестирования на базе **Vitest** (с `jsdom`). Ранее использовался Jest, но проект мигрировал на Vitest для лучшей интеграции с Vite и увеличения скорости сборки.
Тесты предназначены для защиты "швов" архитектуры, проверки математики (статусы, урон) и целостности массивов (hot-path циклы).

## 🚀 Запуск тестов через CMD

Для проверки состояния проекта используйте следующие команды в терминале:

1. **Запуск всех тестов:**

   ```bash
   npm run test
   ```

   *Или напрямую через npx:*

   ```bash
   npx vitest run
   ```

2. **Запуск конкретного тестового файла (например, combat):**

   ```bash
   npx vitest run __tests__/combat.test.ts
   ```

3. **Запуск тестов в режиме Watch (наблюдения):**

   ```bash
   npm run test:watch
   ```

## 🛠️ Мокинг зависимостей (Canvas и Environment)

Многие классы (Tower, Enemy) содержат ссылки на методы `ctx` или `Image` для рендеринга, а некоторые системы (напр. `CollisionSystem`) зависят от объекта `window`.
В `__tests__/setup.ts` добавлены базовые заглушки `HTMLCanvasElement.prototype.getContext`, глобального `Image` и `performance.memory`. При написании интеграционных тестов с отрисовкой используйте моки Canvas API.

> **Внимание, JSDOM!** В среде `jsdom` глобальные переменные `window.innerWidth` и `window.innerHeight` могут быть равны нулю или не инициализированы. `SpatialGrid` инициализируется, опираясь на размер окна. Если в тестах враги не попадают в радиус атаки, убедитесь, что вы замокали размеры окна *до* инициализации `CollisionSystem`:
>
> ```typescript
> Object.defineProperty(window, 'innerWidth', { writable: true, configurable: true, value: 1024 });
> Object.defineProperty(window, 'innerHeight', { writable: true, configurable: true, value: 768 });
> ```

### Ограничения Canvas API в Vitest (jsdom)

Не полагайтесь на полное соответствие Canvas API реальному браузеру в среде `jsdom`.

**Частые проблемы:**

1. **`TypeError: ctx.setLineDash is not a function`**: JSDOM не реализует метод `setLineDash`. Если вы используете его в рендере (например, для отрисовки путей на миникарте), оборачивайте вызов в проверку:

   ```typescript
   if (ctx.setLineDash) ctx.setLineDash([5, 5]);
   ```

2. **Performance Spikes**: В тестах производительности, массированные вызовы `ctx.fillRect` (даже замоканного контекста) могут занимать дольше 16мс, ломая Smoke-тесты. Оптимизируйте циклы отрисовки — например, заливайте фон карты одним большим `fillRect`, а не рисуйте сетку 100x100 тайлов поштучно.

## ⚔️ Интеграционное тестирование (Combat)

Для тестирования комплексных цепочек без Canvas используется `__tests__/combat.test.ts`.
Важно: башня не стреляет сама по себе в `update`. Она только наводится (`TargetingSystem`). Стрельба и спавн пуль происходит исключительно внутри `WeaponSystem.update()`. `dt` должен быть малым (например, `1/60`), чтобы избежать туннелирования (пролетания пули сквозь врага).

## 🛡️ Minimum Viable Testing (MVT) для механик

Просто покрыть 100% строк недостаточно (особенно в GameDev). В проекте требуется написание **негативных тестов** и тестов на краевые случаи производительности и логики. Каждая новая механика эффектов или сущностей обязана сопровождаться такими тестами.

### 1. Тестирование защиты от Double-Triggering (Zombie Events)

Вас всегда должен интересовать вопрос: "А что, если применить эффект к мертвецу в том же кадре, до очистки GC?"
Тест должен набить мертвому врагу урон и убедиться, что эмиттер событий смерти или спавна лута отработал строго 1 раз:

```typescript
it('should trigger death strictly ONCE even on concurrent DOT ticks', () => {
    // Убить (hp = 0)
    // Прокрутить update
    // Проверить toHaveBeenCalledTimes(1)
});
```

### 2. Тестирование Глубины Стека Вызовов (CallStack Depth)

Все AoE / Цепные молнии / Взрывы при смерти могут вызвать рекурсию (враги стоят близко друг к другу и убивают друг друга взрывами).
Вы обязаны написать тест на скопление:

```typescript
it('should safely process explosive chain reactions without crashing', () => {
    // Заспавнить 20 врагов в координате x:100, y:100 с 1 ХП
    // Каждому добавить onDeath взрыв
    // Нанести урон одному
    expect(() => collisionSystem.update()).not.toThrow();
    // Проверить, что все мертвы
});
```

### 3. Стресс-тесты на аллокации (Multi-Stack Testing)

Если вы пишете стакинг аур (например, Slow), протестируйте сценарий наложения десятков стаков, чтобы убедиться, что нет утечек памяти или нелинейных багов:

```typescript
it('should handle 100 concurrent slow applications, keeping only the max', () => {
    for (let i = 0; i < 100; i++) enemy.applySlow(...);
    expect(enemy.slowPower).toBe(expectedMax);
});
```
