# 🏗️ Прагматичная Архитектура (Pragmatic Architecture Rules)

Этот документ описывает философию принятия архитектурных решений в проекте NewTower, особенно на поздних стадиях разработки (Production Phase).

## 1. Hack vs Over-Engineering (Выбор простых решений)

Система должна оставаться максимально простой. Если новая фича требует написания интерфейсов, фабрик и паттернов Strategy для изменения 1-2 строк логики — **выберите простое решение (Hack)**.

**Пример (Модификаторы поведения башен):**
Вместо создания `IAttackBehavior` и переписывания `WeaponSystem` для поддержки отскока снарядов или сплеша:

- **❌ Плохо (Over-engineering):** Внедрение паттерна Strategy, который ломает синхронность базовых систем стрельбы и отдачи.
- **✅ Хорошо (Pragmatic):** Использование конвейера эффектов `Projectile.effects`. Вы пробрасываете примитивный конфигурационный объект через существующий пайплайн стрельбы и обрабатываете его `if/switch` блоками в финальной точке (например, в `CollisionSystem`).

## 2. LOD UI (Сверхдешевые интерфейсы)

В играх жанра Tower Defense отрисовка тысяч элементов UI (текста, иконок) убивает FPS.

- **❌ Плохо:** Рендер текста `🔥 Burn` или HD-иконок над каждым дебаффнутым мобом каждый кадр.
- **✅ Хорошо:** Рендер примитивов (крошечных прямоугольников `ctx.fillRect`) поверх HP-бара (например, Красный = Горение, Синий = Замедление). Это дает абстрактный визуальный фидбек с нулевым влиянием на Draw Calls и Performance.

## 3. Масштаб Алгоритмов (N слишком мало для сложных структур)

Сложные алгоритмические структуры оправданы только на больших объемах данных.

**Пример (Сортировка целей для башни):**
Если башне нужно найти цель с максимальным здоровьем (`healthiest`):

- **❌ Плохо:** Использование `O(N log N)` сортировки или поддержание сложных фильтров в `SpatialGrid`.
- **✅ Хорошо:** Простой `O(N)` линейный поиск (`for` цикл) по массиву врагов *уже находящихся в радиусе* (обычно $N \le 20$). Для таких малых массивов линейный поиск выполняется за доли миллисекунды и бьет любые сложные абстракции.

## 4. Визуальные Индикаторы Состояний (Tint vs Complex UI)

Если нужно показать состояние сложного объекта (например, перегрев Минигана):

- **❌ Плохо:** Создавать новые `StateBar` компоненты, парсить стейт-машины и нагромождать HTML/Canvas UI поверх игровых объектов.
- **✅ Хорошо:** Использовать графические примитивы или альфа-блендинг прямо в методе отрисовки спрайта (например, пульсирующий `ctx.globalAlpha` красный круг поверх спрайта башни или `ctx.filter/compositeOperation` для тонирования). Это читается периферийным зрением и не требует новых слоев логики.

Внедряйте эти правила каждый раз, когда оцениваете риск поломки существующих стабильных систем ради новой фичи.
