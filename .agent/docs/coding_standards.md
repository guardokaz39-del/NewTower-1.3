# 📐 Coding Standards (Best Practices)

В этом документе описаны строгие правила написания кода для проекта **NewTower**, основанного на Entity-Component-System (ECS) подходе и строгом ООП.

## 1. Разделение Ответственности (Separation of Concerns)

Системы не должны выполнять чужую работу. Четко разделяйте:

* **TargetingSystem**: Только ищет цель и поворачивает башню. Никакой стрельбы.
* **WeaponSystem**: Спавнит снаряды (`Projectile`).
* **CollisionSystem**: Наносит урон, запускает эффекты по области, спавнит частицы.

*Почему это важно:* Если башня сама наносит урон без пули (или стреляет внутри `TargetingSystem`), вы ломаете пайплайн модов (карточек) и визуальные спецэффекты.

## 2. Неявная Передача Контекста (Data Primivites vs Objects)

**❌ Плохо:** Передавать весь объект снаряда (`Projectile`) или башни в методы врага, чтобы нанести урон.

```typescript
enemy.takeDamage(enemy, currentProjectile); // ЖЕСТКАЯ СЦЕПКА
```

**✅ Хорошо:** Передавать только примитивы (ID, значения урона, битовые маски `Tags`).

```typescript
enemy.takeDamage(projectile.damage, projectile.sourceTowerId, projectile.sourceCardId, DamageTags.SPLASH);
```

*Почему это важно:* Ссылки на объекты-снаряды, которые могут быть удалены пулом на следующем кадре, приводят к зомби-ссылкам и крашам при попытке вытащить `projectile.x`, когда снаряд уже "переработан".

## 3. Защита от Zombie Events (Zero-HP Guard)

Все публичные методы, мутирующие состояние сущности, **обязаны** проверять ее жизнеспособность первым делом.
Враг, чье здоровье упало в 0, фактически мертв, даже если Garbage Collector/Pool Manager еще не удалил его из списка обновления в этом же кадре.

```typescript
// ✅ Обязательный паттерн для всех методов (урон, замедление, горение, и т.д.)
public applySlow(...) {
    if (!this.isAlive()) return;
    // ... logic
}
```

## 4. Ограничение Рекурсий (Deferred Queue Pattern)

Любое действие, которое может запустить цепную реакцию (А убивает Б, Б убивает В), **строго запрещено** к синхронному исполнению вложенными вызовами `this.die() -> collision.update() -> this.die()`.

Для этого мы используем очереди:

1. Записываем намерение в статический массив (`CollisionSystem.queueExplosion`).
2. В основном цикле `update()` системы делаем `while`-разбор этой очереди.
3. Обязательно вводить лимит (`processingLoops < 10`), чтобы избежать зависания фрейма навсегда.

## 5. Запрет на O(N) циклы внутри `render()` или `update()`

Вы не можете вызывать `array.filter`, `array.find` по огромным коллекциям каждый кадр (60 раз в секунду).
Всё, что является статичным, должно быть **кэшировано** при инициализации. Всё, что динамично, должно использовать `SpatialGrid` или прокэшированные битовые маски.

```typescript
// ❌ Плохо (каждый кадр)
const nearby = allEnemies.filter(e => Math.hypot(e.x - my.x, e.y - my.y) < 100);

// ✅ Хорошо
const nearbyCount = spatialGrid.queryInRadius(my.x, my.y, 100, bufferArray);
```

## 6. TypeScript Strictness (Union Types > Strings)

Никогда не используйте обычный `string` для полей, которые имеют фиксированный набор допустимых состояний (особенно в конфигурациях, модификаторах карт или стейтах). Это приводит к тихим ошибкам и опечаткам.

**❌ Плохо:**

```typescript
interface ICardModifiers {
    targetingMode?: string; // Можно случайно написать 'fiirst' или 'random'
}
```

**✅ Хорошо:**

```typescript
interface ICardModifiers {
    targetingMode?: 'first' | 'last' | 'closest' | 'strongest' | 'healthiest';
}
```

Это гарантирует покрытие всех возможных случаев в `switch` блоках (Exhaustiveness Check) и позволяет редакторам кода подсвечивать ошибки до компиляции.
