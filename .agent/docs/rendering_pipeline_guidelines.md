# Процесс рендеринга и Оптимизации (Rendering Pipeline Guidelines)

Проект использует ванильный Canvas API (2D контекст), что требует тщательного баланса между гибкостью и производительностью. Данный документ описывает правила по работе со слоями и оптимизацией отрисовки.

## 1. Концепт "Baking" (Запекание)

Canvas API экстремально быстр при копировании пикселей (`ctx.drawImage`) и очень медленен при векторной математике, градиентах и фильтрах на каждом кадре.

- **Спрайты врагов:** Запекаются единожды через `SpriteBaker` (см. `sprite_caching.md`). Отрисовка происходит через дешевый `ctx.drawImage`.
- **Статика Карты:** Статические тайлы и объекты (`rock`, `tree`) рисуются ОДИН раз на отдельный Offscreen Canvas в функции `Map.prerender()`. `Map.draw(ctx)` просто копирует этот закэшированный слепок на экран.

> **Правило:** Любой статический, неподвижный контент обязан кэшироваться. Никогда не итерируйте статические объекты каждый кадр в основном цикле отрисовки.

## 2. Глубина (Depth/Z-Sorting Layering)

Движок не имеет полноценного Scene Graph, поэтому "эффект глубины" (2.5D, Isometric, Top-down pseudo-3D) достигается за счет порядка отрисовки (`Painter's Algorithm`).

### Строгий порядок слоев

1. **Background:** Закэшированный статичный слой `Map.draw` (Земля, камни) + `drawAnimatedTiles` (вода, лава).
1b. **Glow Layer:** `Map.drawEndpointGlows` — пульсирующее свечение под порталом/базой (до Z-sort!).
2. **Decals:** Следы слизи, кислотные лужи (`AcidPuddleSystem`).
3. **Z-Sorted Entities:** Совмещенный массив врагов, динамических объектов и **Map эндпоинтов** (портал, база):
   - Если объект имеет высоту (например, Враг, Башня, Мерцающий Кристалл), он должен рисоваться ПОСЛЕ тех, кто стоит физически "выше" по оси Y (т.е. координата Y меньше).
   - *Пайплайн:* Собираем массив `{ y, render: (ctx) => void }` -> Делаем `sort((a,b) => a.y - b.y)` -> Рисуем в цикле.
4. **Air Layer:** Снаряды, партиклы (`EffectSystem`).
5. **Weather/Lighting:** Туман (`FogSystem`), Освещение (`LightingSystem`).
6. **Emissive Elements:** Светящиеся глаза врагов (поверх тумана).
7. **UI:** Дебаг инфо, выделения, хелсбары.

## 3. Динамические эффекты и CPU Thrashing

### Радиальные Градиенты

🚨 **ОПАСНОСТЬ:** Создание `ctx.createRadialGradient` внутри `draw()` на каждый кадр для каждого объекта (например, 50 мерцающих кристаллов) вызовет катастрофическое падение FPS. V8 не может эффективно собирать мусор от тысяч объектов CanvasGradient.

✅ **РЕШЕНИЕ:** Используйте `AssetCache.getGlow(color, size)`. Эта утилита создает градиент ОДИН раз, запекает его в маленький квадратный Offscreen Canvas и возвращает ссылку.
Пульсацию делайте через изменение прозрачности перед копированием спрайта:

```typescript
const glowSprite = AssetCache.getGlow('rgba(0, 255, 255, 1)', 60);
ctx.save();
ctx.globalAlpha = 0.3 + Math.sin(time) * 0.3; // Анимация пульса без аллокаций памяти!
ctx.drawImage(glowSprite, x, y, size, size);
ctx.restore();
```

### Фильтры

🚨 **ОПАСНОСТЬ:** Использование `ctx.filter = 'brightness(2)'`.

✅ **РЕШЕНИЕ:** Используйте альфа-режимы наложения (composite modes) поверх существующих спрайтов или рисуйте белые полупрозрачные прямоугольники.

## 4. Culling (Отсечение невидимого)

Для систем с тысячами объектов (партиклы, враги) всегда используйте примитивный Culling Area Checker:

```typescript
const cullMargin = 64; 
if (e.x < -cullMargin || e.x > viewW + cullMargin || 
    e.y < -cullMargin || e.y > viewH + cullMargin) {
    continue; // Пропускаем отрисовку!
}
e.draw(ctx);
```

Даже если `ctx.drawImage` игнорирует пиксели вне экрана, сам вызов API, вычисление матриц и подготовка состояния имеют накладные расходы. Culling обрезает эти расходы.

## 5. Map State Injection (паттерн из Portal/Base)

Когда нужно реактивно менять визуал Map в зависимости от GameState (волны, HP) — **не создавать Entity**. Использовать:

1. **Примитивные поля** в `Map.ts` (`_waveActive`, `_livesRatio`)
2. **Сеттеры** как публичный API (`setWaveActive()`, `setLivesRatio()`)
3. **Подписки через EventBus** в `GameScene` (не в Map!), пушим в `unsubs`

Преимущества: 0 новых файлов, 0 lifecycle-проблем, EditorScene не затрагивается.
